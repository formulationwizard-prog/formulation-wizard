# Product Packet Architecture — Scoping Memo

**Author:** CC, 2026-05-25 (during operator URL verification pass; parallel to Base/Batch architecture scoping)
**Purpose:** Scope the per-product organizational structure ("Packet" / "Folder" / "Dossier") that holds every regulatory deliverable + artifact the platform generates across a formulation's lifecycle. Output is architectural scope for the next Opus + co-founder strategic session — not implementation.
**Audience:** Next Opus architecture session + co-founder routing call. Informs hierarchy, schema delta, naming, export format, sequencing relative to launch-blocker #4 (Supabase save backend).
**Status:** Scoping memo — no code shipped from this work.

---

## TL;DR

The platform currently generates a substantial body of regulatory + operational content per formulation (label panels, framework determinations, allergen statements, batch sheets, audit trails) — but **none of it persists per-product across sessions**. Save backend is unbuilt (launch-blocker #4). Every generated artifact is re-derived from in-memory state on each render. The Saved tab is a working list but holds only the formulation snapshot, not its generated descendants.

The Packet concept is the **operator-owned product workspace that holds both halves** — platform-generated artifacts (Base Sheet versions, Batch Sheets per production run, generated regulatory deliverables, framework determinations, audit trail) AND operator-input artifacts (PA letters, vendor relationships, packaging structured data, marketing artwork, certifications, lab results, operator notes, source documents). The platform generates what it can determine; the operator brings what the platform can't know. Per [[catalog-must-be-coa-spec-sheet-anchored]] + memory #22 doctrine, both halves persist together per product, with appropriate inheritance from operator/brand-level defaults where applicable.

**Three converging architectural needs make this load-bearing right now:**

1. **Operator's "world-class" framing** 2026-05-25 — customer owns own packaging, own density, own serving size, own pricing. The platform's job is to generate determinations + deliverables and organize them per product. Cost/Sourcing/Packaging recommendations on the formulation page are redundancies; the Packet is where the **outputs** live, separate from the **authoring** surface.

2. **Base/Batch implementation** (separate concurrent scoping memo) — Base Sheet (MMR) + Batch Sheets (BPRs) per 21 CFR 111.205/255 are the regulatory anchor. The Packet is the user-facing organizing structure that holds Base Sheet versions + Batch Sheet history + everything derived from them.

3. **Launch-blocker #4 unblock** — Supabase save backend, when wired, needs to know the persistence shape. Defining the Packet now lets the save backend implementation target the right schema instead of just persisting today's `SavedFormulation` (which would lock in a flat shape inadequate for the artifact inventory below).

**Recommended sequencing:** Architectural decisions (this memo's §7 routing questions Q1–Q8) → schema delta (introduces `OperatorProfile` + `Packet` interfaces) → save backend implementation (launch-blocker #4) → OperatorProfile capture + Packet UI library (replaces Saved tab) → export bundle generation including PA-review packet curation. Approximately **6-8 weeks** of focused work post-decision (revised upward from initial 4-6 estimate after operator-input reframe added operator-level workspace surface); can layer with Base/Batch implementation since they share schema dependencies.

**Recommended naming:** "Packet" — matches existing PA-review-ready packet terminology in memory + industry use for compliance doc bundles. Alternatives evaluated in §6.

## Strategic Context

Operator-stated 2026-05-25 (during the Glucosamine verification pass + capsule diagnostic redundancy discussion):

> "User should input their own packaging as most world class manufacturers have their own custom packaging anyway. User needs to figure out their own density which determines fill weight and choose their own serving size. In the spirit of minimizing the redundancies we should think about that as it pertains to where every artifact lives after a product has been developed. Pricing is a redundancy until the customer inputs their own pricing for ingredients and packaging so economics shouldn't live on the formulation page."

> "Need to create a section for the final packet of material findings and determinations to live per formula. User should be able to have each product in a folder that contains all of the information generated by the Formulation Wizard."

Two distinct architectural directions emerge:

**Direction A — Authoring surface simplification.** Remove redundancies from the formulation-page surface (per-page packaging recommendations, platform-asserted pricing tiles, finished-product density derivation). Push operator-supplied authoritative inputs to the operator. Platform's role on authoring becomes: ingredient composition + regulatory determinations + generated deliverables.

**Direction B — Outputs organization.** Build a per-product container for **everything generated about the product** across its lifecycle. Multiple Base Sheet versions (immutable MMRs), multiple Batch Sheets (per production run), generated regulatory deliverables, framework determinations, PA-review artifacts, attached source documents, export bundles. This is the Packet.

This memo scopes **Direction B**. Direction A is implementation work that lands as authoring-surface cleanup commits (likely in Base/Batch implementation series). The two are complementary: simplifying authoring (A) reduces what lives in the SavedFormulation root; building the Packet (B) gives lifecycle artifacts a proper home.

**Operator-reframe 2026-05-25 (mid-memo-draft):** The Packet is NOT just an archive of platform-generated artifacts — it's an **operator-owned product workspace** that holds both halves. The platform generates what it can determine; the operator brings what the platform can't know. This reframing affects §2 (artifact inventory needs a dedicated operator-input subsection), §4 (schema delta must accommodate operator-level vs product-level placement with inheritance), and §6 (adds permission-model + per-operator-vs-per-product routing question). Per [[catalog-must-be-coa-spec-sheet-anchored]] + memory #22 doctrine, the platform's role is to be authoritative on what it can determine and accommodating of operator-supplied authority on what it can't.

**External developer trial 2026-05-25 (pre-commit validation):** First non-operator user tested the workspace + created three formulations. Captured verbatim in [[external-developer-trial-2026-05-25]]. One quote anchors this memo's urgency:

> "I lost my saves! Anything I made is gone! I guess you need session management."

The tester's experience validates launch-blocker #4 (Supabase save backend) as not-hypothetical-anymore. The Packet schema this memo scopes is downstream of #4 being wired — but the load-bearing case for #4 is now first-hand-user-reported, not architecturally-projected. The same trial surfaced an information-overload finding that motivates Q9 in §6 (step-by-step wizard pattern) + a bottom-line-value gap captured as a post-launch strategic note in §3.

#### 2.7.J Sub-Investigation: How operator-input artifacts plug into the architecture

For each category above, the architecture must address:

**(a) Storage model** — structured fields vs attached files vs hybrid:
- Structured fields: vendor relationships, internal part numbers, packaging dimensions, expiration dates.
- Attached files: PA letters, lab reports, certifications, SOPs, brand artwork, photos.
- Hybrid: certifications = structured metadata (cert type, issuer, expiration) + attached PDF + renewal workflow.

**(b) Versioning semantics** — when an operator-input artifact changes:
- New PA letter → append to PA correspondence history; does NOT bump Base Sheet version.
- Updated vendor pricing → append to vendor history; does NOT cascade to prior Batch Sheets (those captured the lot/price at the time).
- Renewed certification → update in place with renewal-event audit log; expiration tracking restarts.
- Operator note update → append to note history per artifact (notes are append-only per audit doctrine).
- Default rule: operator-input artifacts append/update in place; Base Sheet versions remain pinned to platform-generated state at version-lock time.

**(c) Expiration / renewal handling** — many operator-input artifacts have expiration dates (PA letters, certifications, insurance, lab results). Packet must:
- Surface expiration state per artifact in Packet UI.
- Warn N days before expiration (operator-configurable; default 90 days).
- Support renewal workflow: upload renewed version, retain old version in history, restart expiration tracking.
- Drive export-gate logic: expired certification claimed on label = export gate refusal.

**(d) PA-review impact** — which operator-input artifacts feed into PA-review packet:
- ALWAYS INCLUDE: PA correspondence (prior letters), finished product specs, lab results, allergen control program (if cited), HACCP plan (if PA-reviewable category).
- USUALLY INCLUDE: certifications (if claims depend on them), supplier qualification documents.
- USUALLY EXCLUDE: vendor pricing, internal part numbers, brand artwork, marketing materials, production scheduling.
- OPERATOR-DECISION: photos, operator notes (depends on review context).
- Default inclusion rules per artifact category; operator-overridable per PA-review packet generation.

**(e) Per-operator vs per-product placement (LOAD-BEARING ARCHITECTURAL DECISION)** — some artifacts live at the operator/brand level, others at the product level:

| Artifact | Natural level |
|---|---|
| Insurance certificates (product-liability) | Operator-level (covers all products) |
| Facility allergen control program | Operator-level (facility-specific, not product-specific) |
| GMP / SQF / BRC / FSSC 22000 certification | Operator-level (facility certification) |
| SOP library (cleaning, sanitation, equipment) | Operator-level (facility procedures) |
| Brand guidelines | Brand-level (between operator and product) |
| Co-manufacturing master agreement | Operator-level |
| Quality agreement with supplier | Operator-level |
| Continuing guaranty | Operator-level |
| Kosher / Halal / Organic certifications | Mixed — facility cert is operator-level; per-product cert is product-level |
| PA letter for THIS product | Product-level |
| Finished product spec for THIS product | Product-level |
| Vendor relationship (e.g., Albion TRAACS account) | Operator-level (account terms apply across products) — but per-batch lot purchase is product/batch-level |

## §4. Architecture Proposal

### 4.1 Schema delta

**Option A — Extend `SavedFormulation` further** (minimal change). Add fields directly to the existing interface: `batchSheets?: BatchSheet[]`, `attachedDocuments?: AttachedDocument[]`, `printHistory?: PrintRecord[]`, `formatOverrideHistory?: FormatOverrideRecord[]`. Keeps the top-level shape recognizable.

Pros: minimal type churn; existing UI keeps working.
Cons: `SavedFormulation` becomes a sprawling god-object; difficult to evolve `BatchSheet` lifecycle independently; couples authoring schema (Base Sheet) to lifecycle schema (Packet).

**Option B — Introduce top-level `Packet` interface** (recommended). `SavedFormulation` becomes the Base Sheet content (the authoring product). `Packet` becomes the lifecycle container.

```ts
interface Packet {
  id: string;
  baseSheet: SavedFormulation;       // the formulation + versions
  batchSheets: BatchSheet[];          // production runs against this product
  attachments: AttachedDocument[];   // spec sheets, COAs, photos, reference docs
  printHistory: PrintRecord[];        // every print/export event with timestamp + version + overrides
  formatOverrides: FormatOverrideRecord[]; // Allergen A↔B / FALCPA-exempt operator decisions per product
  generatedSnapshots: GeneratedArtifactSnapshot[]; // per-version frozen deliverables (NFP/SFP/Allergen/etc.)
  reviewArtifacts: PAReviewArtifact[]; // PA verdict capture, signatures, comments
  metadata: PacketMetadata;           // operator notes, custom fields, photos at packet level
}
```

Pros: clean separation of concerns (authoring vs lifecycle); BatchSheet lifecycle independent of Base Sheet versioning; Packet shape evolves without churning authoring schema.
Cons: more schema; migration path for existing `SavedFormulation` users (today: zero users since no persistence — migration is trivial).

**Recommendation:** Option B. The clean-separation benefits outweigh the schema cost. Defer to Opus + co-founder.

**Operator-level extension (post-reframe):** Per §2.7.J(e) per-operator vs per-product placement, Option B's `Packet` interface is the product-level container; a sibling `OperatorProfile` (or `OperatorWorkspace`, `Brand`, etc.) holds operator/brand-level artifacts that inherit down to all Packets owned by that operator. Sketch:

```ts
interface OperatorProfile {
  id: string;
  legalName: string;                                  // operator's legal entity
  doingBusinessAs?: string;                            // brand or DBA name
  facilityAddresses: FacilityAddress[];                // multiple facilities possible
  insuranceCertificates: AttachedDocument[];           // product-liability etc.
  facilityCertifications: FacilityCertification[];     // GMP, SQF, BRC, FSSC 22000 — with expiration
  facilityAllergenControlProgram?: AttachedDocument;   // facility-wide
  sopLibrary: AttachedDocument[];                       // cleaning, sanitation, equipment SOPs
  coManufacturingAgreements: AttachedDocument[];
  qualityAgreements: AttachedDocument[];                // with suppliers
  continuingGuaranties: AttachedDocument[];
  vendorRelationships: VendorRelationship[];           // structured per-operator vendor accounts
  brandGuidelines?: AttachedDocument;
  operatorNotes: OperatorNote[];                        // free-text, append-only history
  // Inheritance defaults — these get pulled into every Packet owned by this operator
  // unless the Packet's own override is set
}

interface Packet {
  id: string;
  operatorId: string;                                   // FK to OperatorProfile (owns the Packet)
  baseSheet: SavedFormulation;                          // formulation + versions
  batchSheets: BatchSheet[];                            // production runs
  attachments: AttachedDocument[];                      // product-level: PA letters, lab reports, FPSS, photos, label artwork
  packagingStructured: PackagingRecord[];                // structured per-product packaging data (replaces packagingName/closureName free-text)
  productCertifications: ProductCertification[];        // per-product certs (Kosher/Halal/Organic on this product)
  printHistory: PrintRecord[];
  formatOverrides: FormatOverrideRecord[];
  generatedSnapshots: GeneratedArtifactSnapshot[];
  reviewArtifacts: PAReviewArtifact[];
  paCorrespondence: PALetterRecord[];                    // PA letters with expiration
  customerLabelingDecisions: CustomerLabelingRecord[];   // which retailers get which version
  productNotes: OperatorNote[];                          // product-level free-text notes
  metadata: PacketMetadata;
}
```

Inheritance rule: PA-review packet generation for a Packet pulls operator-level cert/SOP/insurance + product-level PA letter/spec/lab results. Schema validation enforces that critical operator-level items (e.g., facility-wide allergen control program) exist before product-level Packet review is permitted.

### 4.2 Hierarchy at MVP

Three candidates:

- **Flat — Product → Version → Batch.** No brand grouping. Simple. Matches today's Saved-tab flat shape.
- **Branded — Brand → Product Line → Product → Version → Batch.** Industry-aligned. Requires Brand field decision (queued).
- **Tag-based — Product (with tags) → Version → Batch.** Uses existing `tags?: string[]` field. No new hierarchy; folder-like grouping is operator-defined via tags.

**Recommendation:** Flat at MVP with tag-filtering for organizational flexibility. Branded hierarchy lands post-launch when Brand schema lands. Defer to Opus + co-founder — strategic decision (does single-brand operator early-adopter cohort need Brand grouping out of the gate?).

### 4.3 Storage implications

Launch-blocker #4 (Supabase save backend) needs to know the schema. Recommended (now reflecting operator-level + product-level split):

**Operator-level tables:**
- `operator_profiles` keyed by `operator_id` (auth.users FK).
- `facility_addresses` FK to `operator_profiles`.
- `facility_certifications` FK to `operator_profiles` (with expiration columns + renewal history).
- `operator_attachments` FK to `operator_profiles` (insurance, SOPs, allergen control program, co-mfg agreements, quality agreements, brand guidelines, continuing guaranties).
- `vendor_relationships` FK to `operator_profiles` (structured: account, MOQ, lead time, payment terms, contacts).
- `operator_notes` FK to `operator_profiles` (append-only history).

**Product-level tables:**
- `packets` FK to `operator_profiles`.
- `formulation_versions` FK to `packets` (append-only).
- `batch_sheets` FK to `packets` (with `base_sheet_version` denormalized for query efficiency).
- `product_attachments` FK to `packets` (PA letters, lab reports, FPSS, photos, label artwork, customer specs).
- `packaging_records` FK to `packets` (structured packaging data per product).
- `product_certifications` FK to `packets` (Kosher/Halal/Organic per product, with expiration).
- `pa_correspondence` FK to `packets` (PA letters with expiration tracking).
- `customer_labeling_decisions` FK to `packets`.
- `product_notes` FK to `packets` (append-only).
- `audit_events` FK to `packets` (print history, format overrides, review transitions — event-sourced).
- `generated_snapshots` FK to `formulation_versions` (frozen deliverables per version).

**Storage backend:** Supabase Storage for attachment files; database stores metadata + storage references.

**Auth:** row-level security policies per §4.4a permission model. Per-operator privacy; PA + retailer roles scoped by explicit sharing grants.

### 4.4 Reference vs copy semantics

- **Catalog data:** referenced via `CatalogSnapshotRef` (already designed). When version-pin variant lands, snapshot is reproducible from catalog version tag. Today's legacy variant means the snapshot is non-reproducible — packet ages out of catalog drift.
- **Spec sheets (raw material):** attached to catalog ingredients (canonical location) AND referenced in Packet attachments (per-product convenience). When per-batch COA is needed, attached directly to BatchSheet (specific to that production lot).
- **Operator-level artifacts (insurance, facility certs, SOPs, vendor relationships):** lives in `OperatorProfile` (single canonical record per operator); referenced from Packets that need them; PA-review packet generation pulls them via reference. NEVER copied into product-level Packet (would require operator to maintain N copies).
- **Operator-level → product-level inheritance:** Packet's effective state = (Packet.own fields) ∪ (OperatorProfile.inheritable defaults). Override semantic: when Packet.field is set, it replaces OperatorProfile.field for that product. UI surfaces "inherited from operator" vs "overridden on this product" indicators.
- **PA correspondence:** product-level (PA letters address specific products). Each PA letter is an append-only `PALetterRecord` with expiration tracking + renewal workflow.
- **Vendor relationships:** operator-level structured records (terms, MOQ, contacts apply across products). Per-batch lot purchases reference the vendor relationship + capture lot-specific metadata in BatchSheet.
- **Certifications:** mixed — facility-wide certs (GMP/SQF/BRC/FSSC 22000) are operator-level; per-product certs (Kosher/Halal/Organic on THIS product) are product-level. Schema separates `FacilityCertification` from `ProductCertification`.
- **Generated deliverables:** snapshotted per-version in `generatedSnapshots` table. Re-derivation from snapshot data possible; storage of rendered PDF deferred to export bundle generation.

### 4.4a Permission model (new — per §2.7.J(f))

Row-level security on `operator_id`. Within an operator's scope:

- **Operator role:** full read/write access to OperatorProfile + all owned Packets + all BatchSheets.
- **PA role (per customer-owned PA model):** read access to PA-review packet subsets across operator-granted Packets. Specifically:
  - INCLUDED: compliance docs (HACCP, allergen control, facility certs), spec sheets (raw material + FPSS), lab results, framework determinations, allergen statements, ingredient composition, PA correspondence history, batch sheets (read-only).
  - EXCLUDED: vendor pricing, internal part numbers, brand artwork, marketing materials, production scheduling notes, operator's free-text notes (unless explicitly shared per-finding).
- **Admin role (multi-operator co-packer scenarios):** org-wide access; one admin oversees multiple operator accounts under a parent organization.
- **Retailer role (post-launch):** read access to retailer-onboarding packet subset; per-retailer scoping (Retailer A sees only what Operator has shared with Retailer A).

Implementation: Supabase RLS policies keyed on role + operator scope. PA-review packet generation curates the subset at export time per inclusion rules (defaults from §2.7.J(d), operator-overridable per packet).

### 4.4b Operator-level vs product-level UI surface (new)

OperatorProfile gets its own settings/admin surface (likely a "Workspace Settings" or "Operator Profile" tab) — separate from per-product Packet UI. Lifecycle:

- New operator onboards → fills OperatorProfile (legal entity, facilities, certifications, SOPs, vendor relationships, brand guidelines). One-time setup with periodic update workflow.
- Operator creates new product → new Packet inherits operator-level defaults. Operator can override per-product where needed (e.g., this product uses a different supplier, this product has Kosher cert even though most products don't).
- PA-review-packet export for a product pulls operator-level + product-level material per inheritance rules.

UI implementation deferred post-launch beyond MVP minimum (OperatorProfile field capture during initial save backend wire-up).

### 4.6 Export bundle format

- **Zip of separate PDFs** — one PDF per artifact (NFP, SFP, Allergen, Batch Sheet 1, Batch Sheet 2, etc.). Best for retailer onboarding (specific artifacts).
- **Both** — operator-selectable at export time.

### 4.7 Versioning semantics

- **New version triggers:** any change to ingredients, serving size, package size, product class, mode, or finished product density. Auto-bump per semver:
  - PATCH: corrections that don't change the formula (e.g., supplier name fix on ingredient).
  - MINOR: composition tweaks within same class (e.g., percentage adjustments).
  - MAJOR: regulatory-significant changes (class change, allergen profile change, claim eligibility change).
- **Edit-in-place:** copy edits, operator notes, attachment additions, batch sheet creation — these append to the Packet but don't bump Base Sheet version.
- **Batch Sheet references:** pinned to specific `(baseSheetId, baseSheetVersion)` at creation; cannot be re-pointed.

### Q9 — How does the workspace progressively reveal complexity? (reshaped per operator synthesis 2026-05-25)

**Original framing** (pure wizard / tabbed / hybrid) was too narrow. **Reshaped framing** per operator + CC synthesis:

> How does the workspace progressively reveal complexity to operators of different mastery levels, persona types, and entry-point contexts?

**Candidates:**

- **(A) Mode toggle (Novice / Pro)** — defaulted by user history, picking up the existing Pro/Novice tier infrastructure (`lib/copy/`, `lib/hooks/useTier.ts`, bilingual strings — Phase 5 parked but scaffold is there). Novice mode: guided/focused with progressive disclosure (subsumes both wizard and tabbed views). Pro mode: the dense Workbench operator already uses. Toggle persists per user; explicit override available.
- **(B) Tab-based IA within Build** (Identity / Composition / Delivery / Labels / Regulatory) — orthogonal to the mode toggle; both modes benefit from cleaner IA than today's single-scroll.
- **(C) Shell-level first-touch experience** (welcome / first-product flow) above the workspace — guided onboarding lives at a layer ABOVE Build, not within it.
- **(D) Information-grouping middle layer** — within a single view, better grouping (categories visually clustered rather than interleaved), default-collapsed sections that expand on demand. Reduces cognitive load without structural change.
- **(E) Combination** — most likely answer: (A) + (B) + (D) sequenced; (C) lands with Packet UI entry-point shift.

**CC lean** (updated per operator synthesis): (A) + (B) + (D) sequenced over time, with (C) emerging naturally as Packet UI shifts first-touch entry from Build to Packet library. Pure wizard pattern (original Option A) is the wrong frame — collapses to (A) Novice mode which is the same scaffold extended.

**Constraints surfaced (operator + CC, 2026-05-25):**

1. **Save backend (launch-blocker #4) is precondition** — wizard/Novice mode saves progress incrementally; without persistence, no better than today's single-page (validated by external trial #1 lost-saves experience).
2. **Packet UI entry-point shift** — Packet UI work (Q1–Q8) reshapes the first-touch context. Today operator opens app and lands in Build (blank workspace). Tomorrow operator opens app and lands in Packet library, clicks into a Packet, then edits the Base Sheet draft. Guided-mode question may not even live in Build anymore — it may live at Packet-creation step.
3. **Design system memo with 5-category visual encoding (INPUTS / DERIVED VALUES / DERIVED RENDERS / DETERMINATIONS / DIAGNOSTICS) is higher-ROI predecessor** — every primitive ships with its aesthetic stance baked into its defaults per [[joy-of-mastery-brand-philosophy]]. Component-primitive work depends on this categorical encoding being right first.
4. **Novice tier rendering work + mode toggle work = same project** — don't reinvent the Pro/Novice infrastructure that's already half-built in `lib/copy/` + `lib/hooks/useTier.ts`. Strategic session should explicitly route as one project.
5. **Domain validation needed** — different operator personas (CPG founder bootstrapping first product / established manufacturer QA head with 50 SKUs / co-packer process engineer / non-domain-expert evaluator) may have radically different mode needs. The mode-toggle "graduate from Novice to Pro" arc assumes uniformity; reality may be "QA head wants Workbench day one; bootstrapping founder wants Novice forever." Worth asking Nate before locking pattern.
6. **Default-by-user-history mechanic edge cases** — bootstrapping (brand-new user, no history, what default?), migration (existing user when mode toggle ships, what default?), demo (prospect kicking tires 5-min eval, what default?). Not blockers but need scoping when work lands.

**Strategic input needed:** Confirm the (A) + (B) + (D) sequenced approach + Nate persona-validation gating. Once these route, the work has a clean shape; without routing, risk is reinventing Pro/Novice infrastructure or building wizard pattern that doesn't survive Packet UI entry-point shift.

**Timing:** Post-launch-blocker-#4 + post-design-system-memo. Mid-to-late MVP work; co-development with Packet UI implementation per [[joy-of-mastery-brand-philosophy]] principle 2 (mastery is the engine).
